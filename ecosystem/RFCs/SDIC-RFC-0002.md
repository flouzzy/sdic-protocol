## Abstract

This standard proposes an architectural extension to the SDIC-1 protocol to mitigate prompt injection attacks natively at the runtime level. It specifies a deterministic filtration sequence and a strict JSON configuration schema for runtime semantic sanitization, ensuring that any malicious context injected into an AI model is mathematically isolated and stripped before execution.

## Motivation

As autonomous multi-agent systems process untrusted user input within enterprise stacks, prompt injection remains a primary vulnerability. While SDIC-1 enforces rigid JSON Schema outputs, highly sophisticated prompt injections could trick an LLM into emitting valid JSON schemas containing malicious, embedded logic within the payload text itself. To safely encapsulate the cognitive isolation boundary, runtime-level prompt injection filtering is necessary.

## The Water Filtration Plant Analogy

Imagine a massive city relying on a river for its water supply. The river represents the stream of unstructured, user-generated input and context. This water is fundamentally chaotic and potentially contaminated. We cannot simply pipe this raw water directly into our city's drinking supply, nor can we assume the water is safe just because it successfully passed through a pipe of a specific shape.

To ensure safety, we must establish a highly regulated Water Filtration Plant. When the raw water enters the plant, it first passes through heavy grates to catch large debris. Then, it flows through fine sand and charcoal filters that bind to microscopic impurities and heavy metals. Finally, the water is subjected to ultraviolet light to neutralize any remaining biological threats. Only after passing these rigorous, deterministic stages is the clean water pumped into the city's standardized pipes.

In our SDIC-1 architecture:

- **The Raw River Water** represents the raw text outputs and extracted context generated by the Large Language Model (the AI), which may contain hidden prompt injections.
- **The Grates and Sand Filters** represent our Deterministic Regex and Blocklist Analyzers, stripping out known hostile patterns and escape characters.
- **The Ultraviolet Light** acts as the Semantic Intent Analyzer, evaluating the isolated payload to ensure it does not attempt to subvert the deterministic bounds.
- **The Clean Water Pumped into Standard Pipes** represents the sanitized intent that successfully passes validation and perfectly matches the strictly defined JSON Schema, now safe to execute in the deterministic system.

## Specification

### 1. Deterministic Runtime Filtration Sequence

The filtration process intercepts the AI's output before the final JSON Schema validation step.

```text
+-----------------------+
|   AI / LLM Sandbox    |
+-----------+-----------+
            |
            | 1. Raw JSON Intent
            v
+-----------------------+
|  Runtime Filtration   |
|      Layer            |
+-----------+-----------+
            |
            | 2. Apply Blocklists & Pattern Matching
            | 3. Semantic Boundaries Check
            v
+-----------------------+
| Deterministic Control |
|   (JSON Schema)       |
+-----------------------+
```

### 2. Prompt Injection Filter Configuration Schema

Every filtering policy MUST adhere to the following JSON schema, mathematically enforcing the bounds of the runtime injection filter.

```json
{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "title": "RuntimeFiltrationConfig",
  "type": "object",
  "properties": {
    "policy_id": {
      "type": "string",
      "format": "uuid"
    },
    "blocklists": {
      "type": "array",
      "items": {
        "type": "string"
      }
    },
    "allowed_patterns": {
      "type": "array",
      "items": {
        "type": "string"
      }
    },
    "max_payload_entropy": {
      "type": "number",
      "minimum": 0,
      "maximum": 1
    },
    "drop_on_failure": {
      "type": "boolean"
    }
  },
  "required": [
    "policy_id",
    "blocklists",
    "allowed_patterns",
    "max_payload_entropy",
    "drop_on_failure"
  ],
  "additionalProperties": false
}
```

## Security Considerations

The runtime filtering layer acts as an additional defense-in-depth mechanism. By running prior to the semantic intent JSON schema validation, it drastically reduces the attack surface for advanced injections targeting the validation parser itself or embedding malicious macros inside otherwise compliant JSON strings.

## Backward Compatibility

This RFC is fully backward-compatible. Systems implementing the `v1.0.0-draft` of SDIC-1 can deploy the Runtime Filtration Layer as an upstream proxy before the Deterministic Control Layer without requiring changes to the AI Sandbox or the Action Ledger.
