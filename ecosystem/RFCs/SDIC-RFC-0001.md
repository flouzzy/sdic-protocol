## Abstract

This RFC proposes a standardized JSON schema and interaction protocol for the "Action Ledger" component of the SDIC-1 protocol. The standardization addresses the critical architectural gap of multi-agent synchronization in enterprise environments, ensuring deterministic state transitions when multiple autonomous probabilistic models interact.

## Motivation

As enterprise applications shift towards multi-LLM orchestration (e.g., agent-to-agent communication, parallel model inference), a non-standardized intent execution layer becomes a vulnerability. Divergent interpretations of state by different agents can lead to race conditions, unauthorized actions, and broken guardrails.

By enforcing a uniform, cryptographically verifiable schema for the Action Ledger, this RFC ensures that the host application maintains a single, deterministic source of truth.

## Specification

### 1. Ledger Entry Schema

Every intent passed from the deterministic control layer to the Action Ledger MUST adhere to the following JSON schema:

```json
{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "title": "ActionLedgerEntry",
  "type": "object",
  "properties": {
    "entry_id": {
      "type": "string",
      "format": "uuid",
      "description": "Unique identifier for the ledger entry."
    },
    "agent_id": {
      "type": "string",
      "description": "Identifier of the probabilistic agent proposing the action."
    },
    "intent_type": {
      "type": "string",
      "enum": ["READ", "WRITE", "EXECUTE", "TERMINATE"],
      "description": "The category of the deterministic action."
    },
    "payload": {
      "type": "object",
      "description": "Deterministic payload approved by the guardrails layer."
    },
    "timestamp": {
      "type": "string",
      "format": "date-time"
    },
    "cryptographic_signature": {
      "type": "string",
      "description": "Signature generated by the Deterministic Control Layer validating the payload."
    }
  },
  "required": ["entry_id", "agent_id", "intent_type", "payload", "timestamp", "cryptographic_signature"]
}
```

### 2. Synchronization Flow

To prevent race conditions, the Action Ledger acts as the synchronization primitive. Agents do not communicate intent directly to each other; they read from the Ledger.

```text
+----------+          +-----------------------+          +---------------+
| Agent A  |          | Deterministic Control |          | Action Ledger |
+----+-----+          +-----------+-----------+          +-------+-------+
     |                            |                              |
     | 1. Intent: WRITE (Data)    |                              |
     +--------------------------->|                              |
     |                            | 2. Validate & Sign           |
     |                            +----------------------------->|
     |                            |                              | 3. Commit (State N+1)
     |                            |                              |
+----+-----+                      |                              |
| Agent B  |                      |                              |
+----+-----+                      |                              |
     | 4. Read State              |                              |
     +---------------------------------------------------------->|
     |                            |                              | 5. Return State N+1
     |<----------------------------------------------------------+
     |                            |                              |
```

## Backward Compatibility

This RFC extends the initial SDIC-1 draft without breaking existing abstractions. Applications implementing v1.0.0-draft can adopt the standardized `ActionLedgerEntry` schema for all new intent processing without modifying the underlying Isolation Cognitive or Host Application architectures.
