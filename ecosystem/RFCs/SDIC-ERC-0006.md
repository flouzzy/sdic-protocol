## Abstract

This RFC proposes a standardized mechanism to enhance prompt injection filtering at the runtime level within the SDIC-1 framework. It introduces "Cryptographic Nonce Binding," a method ensuring that the context provided by the host application remains deterministically linked to the final intent emitted by the autonomous agent. This prevents sophisticated multi-turn prompt injection attacks from hijacking the agent's cognitive capabilities to forge unauthorized intents.

## Motivation

As autonomous agents gain the ability to parse external data streams (e.g., browsing the web or reading user-submitted files), the risk of indirect prompt injection rises. An attacker might embed a payload within an innocent-looking document that instructs the agent to ignore its original system prompt and emit a forged `Intent_Proposal`.

While the strict Schema validation of the Deterministic Control Layer filters out structurally malformed intents, it cannot detect an intent that is structurally sound but cognitively hijacked (e.g., a structurally valid intent to transfer funds to the attacker). We need a runtime mathematical proof that the agent is responding to the legitimate context, not an injected subversion.

## The Wax Seal Analogy

Imagine a royal messenger carrying a crucial dispatch from the King to a distant General. The King's letter is sealed with his royal wax signet. Along the journey, an enemy spy intercepts the messenger and attempts to substitute the original orders with a forged letter commanding the General to surrender.

To prevent this, the King introduces a new protocol. Instead of just sealing the envelope, the King embeds a unique, single-use, secret mathematical riddle within the wax of the seal itself. The General, upon receiving the letter, does not just read the orders; he demands the messenger immediately solve the riddle inscribed on the seal. If the messenger presents a letter with a broken seal or cannot solve the specific riddle attached to that exact dispatch, the General immediately recognizes a forgery and rejects the orders, regardless of how authentic the handwriting looks.

In our system:

- **The Royal Messenger** is the autonomous agent.
- **The King's Dispatch** is the legitimate context and system prompt provided by the Host Application.
- **The Forged Letter** is the malicious prompt injection payload trying to subvert the agent's goal.
- **The Unique Secret Riddle** is the `Cryptographic Nonce` generated by the Host Application.
- **Solving the Riddle** is the agent returning the exact `Nonce` symmetrically encrypted or mathematically transformed within its final Intent Schema.
- **The General** is the Deterministic Control Layer verifying the returned Nonce before executing the Intent.

## Specification

To implement Cryptographic Nonce Binding, the communication protocol between the Host Application and the AI Sandbox is updated.

### 1. Context Generation

For every invocation of the AI Sandbox, the Host Application MUST generate a cryptographically secure pseudo-random string (the `nonce`). This `nonce` is embedded securely within the final, un-modifiable segment of the system prompt.

### 2. Intent Schema Extension

The standardized intent schema is extended to require a `runtime_nonce` field. The agent MUST echo the injected `nonce` back within its JSON response.

```json
{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "title": "NonceBoundIntent",
  "type": "object",
  "properties": {
    "intent_action": {
      "type": "string",
      "description": "The deterministic action to execute."
    },
    "runtime_nonce": {
      "type": "string",
      "pattern": "^[A-Za-z0-9+/=]{44}$",
      "description": "The Base64 encoded 256-bit cryptographic nonce provided in the system prompt."
    },
    "payload": {
      "type": "object",
      "additionalProperties": false
    },
    "signature": {
      "type": "string",
      "description": "Cryptographic signature over the canonical representation."
    }
  },
  "required": ["intent_action", "runtime_nonce", "payload", "signature"],
  "additionalProperties": false
}
```

### 3. Canonical Cryptographic Verification

Upon receiving the `NonceBoundIntent`, the Deterministic Control Layer executes the following sequence:

1. **Schema Validation:** Verify the intent matches the strict JSON schema (`"additionalProperties": false`).
2. **Nonce Verification:** Extract `runtime_nonce`. Compare it strictly (constant-time comparison) against the `nonce` generated for that specific execution context. If it does not match exactly, immediately abort and discard the intent as a suspected hijack.
3. **Canonical Signature Verification:** Strip the `signature` field, generate the canonical JSON representation (RFC 8785), compute the SHA-256 hash, and verify the agent's signature using its public key.

## Rationale

Prompt injection payloads typically instruct the model to "ignore previous instructions" and formulate an entirely new response. By enforcing the return of a dynamically generated, high-entropy nonce, we mathematically bind the output to the input context. An injected payload cannot predict the nonce, and forcing the model to ignore its system prompt will simultaneously strip its awareness of the required nonce, resulting in a deterministic verification failure at the control layer.

## Backward Compatibility

This RFC modifies the minimum required intent structure. Upgrading to this standard requires updates to both the Host Application (to generate and verify nonces) and the agent's system prompt (to instruct the inclusion of the nonce). It is a breaking change for systems not currently enforcing dynamic context binding.
